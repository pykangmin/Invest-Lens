# 바이브 코딩 환경 설명서

---

## 전체 구조 개요

이 레포지토리는 AI 에이전트가 코드를 작성할 때 "어디에서 시작하고, 무엇을 만들고, 어떻게 마무리하는지"를 스스로 판단할 수 있도록 설계된 작업 환경이다. 에이전트는 매 세션마다 AGENTS.md를 읽고 현재 상태를 파악한 뒤, feature_list.json에서 다음 할 일을 골라 하나씩 구현하며, 끝나면 진행 기록을 남기고 깨끗한 상태로 다음 세션에 넘긴다.

이 구조는 두 편의 엔지니어링 문서에 근거한다. OpenAI의 "Harness Engineering"은 에이전트가 작업할 환경 자체를 설계하는 방법론이고, Anthropic의 "Effective Harnesses for Long-Running Agents"는 여러 세션에 걸쳐 에이전트가 끊김 없이 작업을 이어가는 방법론이다. 이 두 접근을 결합하여, 투자 데이터 대시보드를 바이브 코딩으로 구현하기 위한 환경을 구성했다.

레포지토리의 모든 파일은 두 가지 층위 중 하나에 속한다.

**하네스 규칙**은 에이전트가 반드시 지켜야 하는 규칙이다. 크게 두 종류로 나뉜다. 첫째는 구조적 규칙으로, 레이어 의존성 방향, 기능 목록의 무결성, 세션 시작/종료 절차가 여기에 해당한다. 둘째는 데이터 해석 규칙으로, skills/ 디렉토리에 담긴 데이터 프로파일링·분석·시각화·인사이트·레이아웃 규칙이 여기에 해당한다. 두 종류 모두 반드시 지켜져야 하며, 구조적 규칙은 verify-structure.sh가, 데이터 해석 규칙은 별도의 검증 메커니즘이 강제한다. Skills는 에이전트가 선택적으로 참고하는 가이드가 아니라, 대시보드 자동 생성의 근거가 되는 규칙 엔진이다.

**해커톤 제출**은 DACON 대회의 산출물 형식 요구사항이다. 에이전트의 코딩 동작에는 영향을 주지 않는다.

## 에이전트의 작업 흐름

에이전트는 새 세션을 시작할 때마다 이전 세션의 기억이 없다. 교대 근무에 투입된 엔지니어가 인수인계 문서를 읽고 상황을 파악하는 것과 같다. 이 레포지토리에서는 그 인수인계를 세 가지 파일이 담당한다: git 커밋 로그는 "코드에 무엇이 바뀌었는지"를, claude-progress.txt는 "이전 세션이 무엇을 했고 무엇이 남았는지"를, feature_list.json은 "다음에 무엇을 해야 하는지"를 알려준다.

### 세션 시작

AGENTS.md에 적힌 순서대로, 에이전트는 먼저 git log와 claude-progress.txt를 읽어 현재 상태를 파악한다. 그다음 init.sh를 실행하여 개발 서버를 기동하고, 서버가 정상 응답하는지 확인하고, verify-structure.sh와 verify-skills.sh로 구조 규칙과 Skills 규칙이 모두 지켜지고 있는지 검증한다. 이 단계들이 모두 통과하면, feature_list.json에서 아직 완료되지 않은(passes: false) 기능 중 우선순위가 가장 높은 항목을 선택한다.

### 기능 구현

에이전트는 한 세션에 하나의 기능만 구현한다. 여러 기능을 동시에 작업하면 컨텍스트 윈도우가 소진되어 어느 것도 완성하지 못하는 실패가 반복되기 때문이다. 선택한 기능과 관련된 skills/ 파일을 읽고, docs/architecture/LAYERS.md의 레이어 규칙에 따라 올바른 디렉토리에 코드를 작성한다.

### 세션 종료

기능을 구현한 뒤에는 docs/guides/TESTING.md의 절차에 따라 검증하고, feature_list.json의 해당 항목을 passes: true로 변경한다. .gitcommit-rules.md의 형식에 맞춰 git 커밋을 남기고, claude-progress.txt에 이번 세션의 작업 요약을 추가한다. 다음 세션의 에이전트가 별도의 정리 작업 없이 바로 새 기능에 착수할 수 있는 상태를 만드는 것이 목표다.

구현 도중 문제가 생기면 git revert로 마지막 정상 상태로 돌아간 뒤 다시 시도한다. 반쯤 완성된 코드를 남기는 것보다, 깨끗하게 되돌리는 것이 다음 세션에 더 유리하다.

## 루트 진입점

### AGENTS.md

레포지토리의 목차 역할을 하는 파일이다. 에이전트가 가장 먼저 읽으며, 다른 모든 파일의 위치와 역할을 약 100줄 이내로 안내한다.

OpenAI는 처음에 모든 규칙을 하나의 파일에 담았으나, 컨텍스트 윈도우에서 지시 파일이 차지하는 공간이 너무 커서 정작 코드 작업에 쓸 공간이 부족해지는 문제가 발생했다. 이를 목차 수준으로 축소하고 상세 규칙은 개별 파일에 분산 배치하는 방식으로 해결했다. "모든 것이 중요하면, 아무것도 중요하지 않다"는 판단이었다.

AGENTS.md에는 다섯 가지가 적혀 있다: 불변 규칙 파일의 위치, 세션 시작/종료 절차 체크리스트, 데이터 해석 규칙(skills/) 파일의 위치, 레이어 구조 문서(docs/architecture/LAYERS.md)의 위치, 기술 스택 문서(docs/tech-stack.md)의 위치. 에이전트는 이 파일을 읽는 것만으로 "어디를 더 읽어야 하는지"를 알 수 있다.

레이어 구조와 의존성 방향은 별도의 선언 파일을 두지 않고, docs/architecture/LAYERS.md에 직접 상세와 매트릭스까지 함께 기술한다. AGENTS.md가 목차, LAYERS.md가 진실의 원천이 되는 2단 참조 구조다. 이는 OpenAI가 강조한 "progressive disclosure" 패턴에 부합한다.

## 하네스 인프라

### init.sh

매 세션 시작 시 실행하는 부팅 스크립트다. 의존성 설치 확인, 개발 서버 기동, HTTP 헬스체크, verify-structure.sh 실행, verify-skills.sh 실행을 순서대로 수행한다. 모든 단계가 통과하면 "환경 준비 완료"를 출력한다.

Anthropic은 에이전트가 새 세션을 시작할 때 가장 먼저 개발 서버를 기동하고 기본적인 동작 확인을 하도록 지시했다. 이전 세션이 앱을 깨진 상태로 남겼다면, 에이전트가 새 기능 구현에 들어가기 전에 즉시 발견하여 수정할 수 있기 때문이다. 깨진 상태를 모른 채 새 기능을 추가하면 문제가 더 깊어진다.

OpenAI도 "한 명령으로 부팅(one-command boot)"을 하네스의 핵심 구성으로 꼽았다. 에이전트가 환경을 세팅하는 방법을 매번 추론할 필요 없이, init.sh 하나만 실행하면 되도록 만드는 것이다.

### verify-structure.sh

docs/architecture/LAYERS.md에서 정의한 레이어 의존성 매트릭스가 실제로 지켜지고 있는지 검사하는 스크립트다. TypeScript 소스 파일들을 파싱하여 import 구문의 경로가 레이어 규칙을 위반하는지 확인한다. 예를 들어 src/analysis/ 디렉토리의 코드가 src/visualization/을 import하고 있으면 에러를 출력하고, 그 에러 메시지 안에 "어떻게 수정해야 하는지"를 함께 알려준다.

이 설계는 OpenAI의 핵심 패턴에서 가져왔다. OpenAI는 커스텀 린터의 에러 메시지에 교정 지시를 포함시켜, 에이전트가 규칙을 위반했을 때 별도의 문서를 찾아보지 않아도 수정 방법을 바로 알 수 있도록 했다. 에러 메시지 자체가 에이전트에게 주어지는 컨텍스트이므로, 그 안에 해결책을 넣으면 수정까지의 경로가 최단이 된다.

문서에 규칙을 적어두는 것과 스크립트로 검증하는 것은 근본적으로 다르다. 문서의 규칙은 에이전트가 읽지 않거나 무시할 수 있지만, 스크립트가 실패하면 에이전트는 그 에러를 해결하지 않고는 진행할 수 없다.

### verify-skills.sh

skills/ 디렉토리의 데이터 해석 규칙이 실제 코드에서 지켜지고 있는지 검사하는 스크립트다. Skills.md는 빌드 시점에 Claude Code에 의해 TypeScript 함수로 변환되는데, verify-skills.sh는 원본 규칙과 생성된 함수 사이의 일치성을 검증한다.

예를 들어 visualization.skill.md가 "시계열 1변수 → 라인 차트"라는 매핑 테이블을 정의하고 있다면, 이 스크립트는 매핑 테이블을 파싱해서 src/visualization/의 차트 선택 함수가 그 매핑을 빠짐없이 반영하고 있는지 확인한다. data-profile.skill.md의 스키마 감지 규칙이 정의한 분류 기준과 실제 데이터 로더의 태깅 로직이 일치하는지도 검사한다.

verify-structure.sh가 "코드의 모양"을 검사한다면, verify-skills.sh는 "데이터 해석의 일관성"을 검사한다. 해커톤 평가 기준에서 Skills.md 설계(25점)와 대시보드 자동 생성(25점)이 모두 "규칙 문서가 실제로 작동하는가"를 본다는 점에서, 이 스크립트는 Skills를 정적 문서에서 살아있는 규칙 엔진으로 격상시키는 핵심 장치다. Skills.md 규칙과 생성된 코드 사이의 대응 관계를 기계적으로 검증함으로써, 심사위원에게 "이 대시보드가 Skills.md에 의해 생성되었다"는 주장의 근거를 제공한다.

### feature_list.json

대시보드에 필요한 모든 기능을 JSON 배열로 나열한 파일이다. 각 항목에는 기능 설명, 검증 절차(steps), 완료 여부(passes: true/false)가 포함된다.

Anthropic은 이 파일을 프로젝트 초기에 생성하도록 했다. 에이전트가 "전체 기능의 범위"를 한눈에 보고, 이미 완료된 기능과 아직 남은 기능을 구분할 수 있게 하기 위함이다. 에이전트가 주변을 둘러보고 "충분히 완성된 것 같다"고 조기에 판단하는 실패를 방지한다 — passes: false인 항목이 남아 있는 한, 프로젝트는 완료가 아니다.

JSON 형식을 사용한 이유는 실용적이다. Anthropic의 실험에서 에이전트는 마크다운 파일의 내용을 자의적으로 수정하거나 삭제하는 경향이 있었지만, JSON 파일의 구조화된 필드는 함부로 변경하지 않았다. 에이전트에게는 passes 필드의 값만 false에서 true로 바꾸도록 지시하며, 항목 자체를 삭제하거나 편집하는 것은 명시적으로 금지한다.

각 항목의 steps 필드는 E2E 검증 시나리오를 겸한다. 에이전트가 기능을 완료로 표시하기 전에, steps에 적힌 절차를 따라 실제로 동작하는지 확인해야 한다.

### claude-progress.txt

세션 간 인수인계를 위한 진행 기록 파일이다. 에이전트는 매 세션 종료 시 "이번 세션에서 무엇을 했고, 어떤 문제가 있었고, 다음에 무엇을 하면 좋은지"를 이 파일에 추가한다. 다음 세션의 에이전트는 git log와 함께 이 파일을 읽어 현재 상태를 빠르게 파악한다.

git log가 "코드 수준"의 변경 이력을 보여준다면, claude-progress.txt는 "의도 수준"의 작업 이력을 보여준다. 커밋 메시지만으로는 "왜 이 접근을 선택했는지", "어떤 시도가 실패했는지"를 알기 어렵지만, 진행 기록에는 그런 맥락이 남는다.

### .gitcommit-rules.md

커밋 메시지 형식과 커밋 타이밍 규칙을 정의한 파일이다. 커밋 메시지는 Conventional Commits 형식을 따른다. `feat(layer): ...` 형식을 기본으로 하며, 버그 수정은 `fix(layer): ...`, 리팩터링은 `refactor(layer): ...`를 쓴다. 괄호 안의 `layer` 위치에는 변경이 속한 영역을 명시한다 — src/의 레이어명(types, schema, data-loader, analysis, visualization, layout, shared), skills/의 파일명, 또는 docs·infra 같은 상위 카테고리 중 하나를 쓴다. 빌드가 깨진 상태에서는 커밋하지 않는다. 구현에 실패하면 git revert로 마지막 정상 상태로 복귀한 뒤 재시도한다.

Anthropic은 에이전트에게 설명적 커밋 메시지와 함께 커밋하도록 지시했다. 이를 통해 잘못된 변경을 되돌리고 작동하던 상태를 복구하는 것이 가능해졌다. 커밋 이력 자체가 에이전트의 작업 로그이자 복구 수단이 되는 셈이다.

## 문서 계층 (docs/)

### docs/tech-stack.md

프로젝트에서 사용하는 기술 스택과 그 선정 이유를 기록한 파일이다. 프레임워크(React + Vite), 언어(TypeScript), 차트 라이브러리(ECharts), 데이터 처리 도구(PapaParse, lodash, dayjs), 배포 플랫폼(Vercel)이 담긴다.

이 파일은 두 종류의 독자를 상정한다. 첫째는 Claude Code 에이전트로, 코드를 생성할 때 어떤 라이브러리를 써야 하는지 이 파일을 참조한다. 둘째는 팀원과 심사위원으로, 프로젝트가 어떤 도구로 만들어졌고 왜 그 도구를 선택했는지 이해하는 데 쓴다. AGENTS.md 목차에서 이 파일을 가리키므로, 에이전트는 세션 시작 시점에 이 파일의 존재와 위치를 알게 된다.

SETUP.md가 "개발 환경을 실제로 세팅하는 방법"을 담는다면, tech-stack.md는 "왜 이 스택을 골랐는가"의 선택 기록을 담는다. 전자는 실무 지시, 후자는 의사결정 문서다.

### docs/architecture/LAYERS.md

시스템의 레이어 구조와 의존성 방향을 정의하는 단일 진실의 원천이다. src/ 디렉토리의 6개 레이어가 어떤 순서로 의존하는지, 각 레이어의 책임 범위는 무엇인지, 다른 레이어와의 허용/금지 의존성은 어떻게 되는지를 매트릭스로 기술한다.

의존성은 Types → Schema → DataLoader → Analysis → Visualization → Layout 순서로, 오직 순방향으로만 흐른다. 예를 들어 Analysis는 Schema를 import할 수 있지만, Visualization을 import할 수 없다. Types 레이어는 외부 의존성이 없고, Schema 레이어는 Types만 import할 수 있으며, DataLoader는 Types와 Schema를 import할 수 있다는 식으로 각 레이어의 구체적인 경계를 정의한다. verify-structure.sh는 이 매트릭스를 기준으로 위반 여부를 검사한다.

OpenAI는 이런 레이어 의존성 규칙이 보통 수백 명 규모의 팀에서나 도입되는 것이지만, 에이전트와 작업할 때는 처음부터 필요하다고 지적했다. 에이전트는 기존 코드의 패턴을 충실히 복제하므로, 초기에 잘못된 의존성이 하나라도 생기면 이후 에이전트가 그 패턴을 반복 확산시키기 때문이다.

원본 OpenAI 문서는 레이어 선언을 루트 AGENTS.md와 별개 문서로 분리하는 2단 참조 구조를 채택했다. 이 레포지토리도 동일하게 "AGENTS.md(목차) → LAYERS.md(상세)" 2단 구조를 따른다. 목차(AGENTS.md)를 짧게 유지하면서도, 필요할 때 상세를 깊이 있게 참조할 수 있게 하는 점진적 공개(progressive disclosure) 패턴이다.

### docs/guides/SETUP.md

개발 환경 세팅 절차를 기술한 파일이다. Node.js 20.x LTS, npm을 기본 패키지 매니저로 삼고, 필수 의존성 설치 명령과 개발 서버 기동 방법을 담는다. 주요 의존성은 React, Vite, TypeScript, ECharts(echarts-for-react), PapaParse, lodash, dayjs이다.

init.sh가 이 문서의 내용을 스크립트로 자동화한 것이라면, SETUP.md는 사람이 읽거나 에이전트가 환경 문제를 디버깅할 때 참조하는 원본이다. tech-stack.md가 "왜 이 스택인가"를 담는다면, SETUP.md는 "이 스택을 어떻게 설치하고 실행하는가"를 담는다.

### docs/guides/TESTING.md

테스트 전략과 검증 절차를 기술한 파일이다. 헬스체크(서버 응답 확인), 구조 검증(verify-structure.sh), Skills 준수 검증(verify-skills.sh), 기능 검증(feature_list.json의 steps 실행), 시각적 검증(스크린샷 비교) 등 수준별 테스트 방법을 정의한다.

Anthropic은 에이전트가 기능을 완료로 표시하기 전에 반드시 사람처럼 E2E 테스트를 수행하도록 지시했다. 이 지시가 없으면, 에이전트는 단위 테스트나 개발 서버에 대한 요청만으로 검증하고 실제로 동작하지 않는 기능을 완료로 표시하는 경향이 있었다.

### docs/exec-plans/ (선택적)

현재 진행 중인 기능의 상세 실행 계획을 active/에, 완료된 기능의 실행 기록을 completed/에 보관하는 디렉토리다. OpenAI가 사용한 패턴이며, 에이전트가 "왜 이렇게 구현하기로 했는지"의 의사결정 맥락을 남길 수 있다.

feature_list.json + claude-progress.txt가 "무엇을 했는가"를 추적한다면, exec-plans/는 "왜, 어떻게 하기로 했는가"를 추적한다. 해커톤 규모에서는 필수가 아니므로 선택적으로 채택한다. 생략하면 의사결정 기록이 빠지지만, 기능 추적 자체는 feature_list.json으로 충분하다.

### docs/references/

사용하는 차트 라이브러리, 데이터 API, 외부 도구의 참고 문서를 보관하는 디렉토리다. 에이전트는 코드를 작성할 때 이 디렉토리의 문서를 참조하여 라이브러리의 올바른 사용법을 확인한다.

초기 캐시 대상은 네 가지다. ECharts의 차트 옵션 레퍼런스, PapaParse의 CSV 파싱 옵션, lodash에서 자주 쓰는 유틸리티 함수, dayjs의 날짜 포맷 토큰. 각각의 공식 문서에서 핵심 페이지를 마크다운으로 덤프해 보관한다.

OpenAI는 에이전트 관점에서 "레포지토리 안에서 접근할 수 없는 지식은 존재하지 않는 것과 같다"고 지적했다. 슬랙 스레드나 외부 문서에만 있는 정보는 에이전트가 활용할 수 없으므로, 필요한 참고 자료는 레포지토리 안에 복사해 두어야 한다. 특히 라이브러리 API는 에이전트가 기억으로부터 잘못 추론할 위험이 크기 때문에, 공식 문서의 사본을 레포 안에 두는 것이 실수 예방에 효과적이다.

## Skills 문서 (skills/)

### 디렉토리의 위상

skills/ 디렉토리는 해커톤에서 제출하는 Skills.md 파일들을 모아둔 곳이다. 디렉토리 이름은 DACON 대회의 요구사항에서, 내부 파일의 작성 형식은 OpenAI의 golden-principles 패턴(각 30~60줄, 결정 가능한 규칙 + 예외 명시)에서 가져왔다.

이 레포지토리에서 skills/는 하네스 규칙의 일부다. 에이전트가 "합리적인 이유가 있으면 다르게 판단할 수 있는" 가이드가 아니라, 반드시 복종해야 하는 규칙 엔진이다. 그 이유는 해커톤 평가 기준에 있다. 범용성(25점)과 Skills.md 설계(25점)는 "규칙 문서가 얼마나 명확하고 일관되게 적용되는가"를 보고, 대시보드 자동 생성(25점)과 바이브코딩 활용(15점)은 "그 규칙이 실제로 코드에 반영되어 동작하는가"를 본다. Skills를 선택적 가이드로 취급하는 순간 이 네 항목(총 90점)이 모두 약해진다.

skills/ 파일의 규칙 준수 여부는 verify-skills.sh가 강제한다. verify-structure.sh가 코드의 구조를 검사한다면, verify-skills.sh는 데이터 해석의 일관성을 검사한다. 둘 다 init.sh의 부팅 과정에서 실행되며, 실패 시 에이전트는 그 에러를 해결하지 않고는 다음 작업으로 넘어갈 수 없다.

아래의 다섯 개 파일은 대회 요구사항이 예시로 제시한 다섯 가지 역할(데이터 분석 기준, 지표 계산 규칙, 시각화 선택 기준, 리포트 구성 흐름, 인사이트 생성 규칙)을 참고하여 구성한 것이다. 실제 구현 시에는 대시보드의 방향에 따라 파일을 추가·통합·재분류할 수 있다. 중요한 것은 파일의 개수나 이름이 아니라, 각 규칙이 에이전트가 검증할 수 있는 형태로 작성되어 있는지 여부다.

### data-profile.skill.md

들어온 데이터를 받아 "이게 어떤 종류의 투자 데이터인지"를 판정하는 규칙을 담는다. 컬럼 타입 추론(datetime, numeric, category, identifier), 분포 특성(고유값 수, 결측 비율, 값의 범위), 관계 구조(단일 계열/다중 계열/횡단면/패널) 등을 보고 분석 카테고리를 결정한다. 예를 들어 "datetime 컬럼 1개 + numeric 컬럼 1개 → 단일 시계열", "category 컬럼(고유값 7개 이하) + numeric 컬럼 → 범주 비교"처럼 데이터 형태와 카테고리의 매핑을 기술한다.

이 파일이 별도로 분리된 이유는 해커톤 평가 기준 1순위인 "범용성 25점" 때문이다. 범용성은 "임의의 투자 데이터가 들어와도 대응 가능한가"를 본다. 스키마가 하드코딩되어 있으면 이 점수는 낮을 수밖에 없다. data-profile.skill.md는 대시보드의 입구 역할을 하며, 나머지 skills 파일들은 이 파일이 반환한 카테고리를 입력으로 받아 동작한다.

### data-analysis.skill.md

특정 분석 카테고리에 해당하는 데이터에 대해 어떤 지표를 어떻게 계산할지를 정의한다. data-profile.skill.md가 "이게 시계열이다"라고 판정하면, data-analysis.skill.md가 "시계열이면 수익률·변동성·이동평균을 계산한다"는 규칙을 적용한다. 지표 계산 공식(수익률, 변동성, 샤프비율 등)과 데이터 유효성 검증 규칙을 담는다. 에이전트가 src/analysis/ 레이어의 코드를 작성할 때 이 파일을 참조한다.

이 파일에서는 "스키마 감지" 책임을 지지 않는다. 스키마 감지는 data-profile.skill.md의 역할이며, 이 파일은 감지 결과를 입력으로 받는다.

### visualization.skill.md

분석 카테고리와 지표 특성에 따른 차트 선택 규칙을 정의한다. "시계열 1변수에는 라인 차트", "범주 비교 7개 이하에는 바 차트"처럼 데이터 타입과 차트 타입의 매핑을 테이블로 기술한다. 이 테이블은 에이전트가 검증 함수(validateChartSelection)로 구현하여 규칙 위반 시 경고를 출력하도록 설계되어 있으며, verify-skills.sh가 이 검증을 실행한다.

문서에 규칙을 적어두는 것에 그치지 않고, 그 규칙을 코드로 검증할 수 있는 형태로 작성하는 것이 핵심이다. "차트를 적절하게 선택한다"는 지시는 에이전트가 해석할 여지가 넓어 일관성이 떨어지지만, 매핑 테이블과 검증 함수는 판단 기준을 명확히 고정한다.

### insight.skill.md

대시보드에 표시할 인사이트(해석, 요약, 경고)의 생성 기준을 정의한다. 어떤 조건에서 어떤 종류의 인사이트를 생성하는지, 인사이트의 표시 형식과 우선순위는 어떻게 결정하는지를 기술한다. 에이전트가 분석 결과를 사용자에게 전달할 때의 규칙이다.

### dashboard-layout.skill.md

대시보드의 UI 구성 규칙을 정의한다. 컴포넌트 배치 순서, 반응형 레이아웃 기준, 데이터 밀도에 따른 레이아웃 분기 등을 담는다. 에이전트가 src/layout/ 레이어의 코드를 작성할 때 이 파일을 참조한다.

## 소스 코드 (src/)

### 레이어 구조

src/ 디렉토리는 docs/architecture/LAYERS.md에서 정의한 레이어 구조를 물리적 디렉토리로 분리한 것이다. 각 레이어는 자신보다 앞선 레이어만 import할 수 있고, 뒤에 오는 레이어를 import할 수 없다.

**types/**는 시스템 전체에서 사용하는 기본 타입을 정의한다. 숫자, 날짜, 통화, 비율 등의 타입이 여기에 속한다. 다른 레이어에 대한 의존성이 없으며, 모든 레이어가 이 레이어를 import할 수 있다.

**schema/**는 투자 데이터의 구조를 정의한다. 종목 정보, 포트폴리오 구성, 재무 지표 등의 스키마가 여기에 속한다. types/만 import할 수 있다.

**data-loader/**는 외부 데이터를 수집하고 파싱하는 코드를 담는다. CSV 파일 읽기, API 호출, JSON 파싱 등의 로직이 여기에 속한다. types/와 schema/를 import할 수 있다. data-profile.skill.md의 스키마 감지 규칙을 실제로 적용하는 레이어가 여기다.

**analysis/**는 지표 계산과 분석 로직을 담는다. 수익률 계산, 변동성 분석, 상관관계 계산 등이 여기에 속한다. types/, schema/, data-loader/를 import할 수 있지만, visualization/이나 layout/은 import할 수 없다.

**visualization/**는 차트와 그래프를 렌더링하는 코드를 담는다. analysis/의 계산 결과를 시각적 요소로 변환한다. visualization/이 analysis/를 import하는 것은 허용되지만, 그 반대(analysis/가 visualization/을 import)는 금지된다.

**layout/**은 대시보드의 최종 UI를 구성한다. visualization/의 차트 컴포넌트들을 배치하고, 사용자 인터랙션을 처리한다. 모든 앞선 레이어를 import할 수 있다.

**shared/**는 레이어가 아니라 유틸리티 모음이다. 여러 레이어에서 공통으로 쓰는 작은 도구들(예: 에러 바운더리, 공통 상수, 타입 가드 등)을 필요할 때 파일 단위로 추가하는 자리다. 모든 레이어가 shared/를 import할 수 있지만, shared/는 다른 레이어를 import하지 않는다. 비즈니스 로직은 절대 들어가지 않는다 — 지표 계산이나 차트 선택 같은 로직이 shared/에 들어오려 하면 그건 해당 레이어(analysis/, visualization/ 등)에 있어야 하는 것이다.

OpenAI 원본은 auth, telemetry, feature flags 같은 교차 관심사를 Providers라는 단일 인터페이스로 격리하는 패턴을 제시했다. 이 레포지토리는 해커톤 규모에서 그 수준의 추상화가 과하다고 판단하여, 레이어 대신 유틸리티 디렉토리로 단순화했다. 실제로 교차 관심사가 등장하는 순간에만 파일을 추가하는 YAGNI 원칙을 따른다.

## 요소 간 참조 관계

이 레포지토리의 파일들은 독립적으로 존재하지 않고, 서로를 참조하며 하나의 작업 흐름을 구성한다. 참조 방향을 정리하면 다음과 같다.

AGENTS.md는 레포지토리의 다른 모든 파일을 참조한다. 목차이므로 당연하다. 특히 레이어 구조는 docs/architecture/LAYERS.md를, 기술 스택은 docs/tech-stack.md를, 데이터 해석 규칙은 skills/ 디렉토리를 가리킨다.

docs/architecture/LAYERS.md는 레이어 구조의 단일 진실의 원천이다. AGENTS.md가 이 파일을 목차에서 가리키며, verify-structure.sh가 이 파일의 의존성 매트릭스를 기준으로 검사를 수행한다.

docs/tech-stack.md와 docs/guides/SETUP.md는 상보적이다. tech-stack.md가 "어떤 스택을 왜 골랐는가"의 의사결정 기록이라면, SETUP.md는 "그 스택을 어떻게 설치하고 실행하는가"의 실무 지시다. init.sh가 SETUP.md의 내용을 자동화한다.

init.sh는 verify-structure.sh와 verify-skills.sh를 호출하고, docs/guides/SETUP.md의 환경 세팅을 자동화한다.

feature_list.json의 각 항목은 skills/ 디렉토리의 특정 파일과 연결된다. 예를 들어 "시계열 차트 렌더링" 기능은 visualization.skill.md와 data-profile.skill.md를, "샤프비율 계산" 기능은 data-analysis.skill.md를 참조하는 식이다. 실제 skill 파일의 구성이 달라지면 이 연결도 함께 바뀐다.

skills/ 파일들은 서로 간에도 순서가 있다. data-profile.skill.md가 데이터의 카테고리를 판정하면, 그 결과를 입력으로 data-analysis.skill.md가 지표를 계산하고, visualization.skill.md가 차트를 선택하고, insight.skill.md가 인사이트를 생성하고, dashboard-layout.skill.md가 최종 배치를 결정한다. 데이터에서 UI까지의 파이프라인이 skills/ 내부에도 그대로 반영되어 있다.

skills/ 파일들은 src/의 특정 레이어와 대응한다. data-profile.skill.md는 src/data-loader/와, data-analysis.skill.md는 src/analysis/와, visualization.skill.md는 src/visualization/과, dashboard-layout.skill.md는 src/layout/과 연결된다. insight.skill.md는 analysis/의 결과를 해석하여 layout/에 전달되므로 두 레이어에 걸쳐 있다. 에이전트가 특정 기능을 구현할 때, 해당 skill 파일을 읽고 대응하는 src/ 레이어에 코드를 작성하는 흐름이다.

verify-skills.sh는 skills/의 규칙과 src/의 실제 구현 사이의 일관성을 검사한다. 예를 들어 visualization.skill.md의 매핑 테이블과 src/visualization/의 차트 선택 로직이 어긋나면 실패한다.

claude-progress.txt와 git 커밋 로그는 시간 순서로 축적되며, 다음 세션의 에이전트가 현재 상태를 파악하는 데 함께 사용된다.
