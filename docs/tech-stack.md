# 대시보드 구현 기술 스택

## 확정 스택

| 층 | 선택 |
|---|---|
| 프레임워크 | React + Vite |
| 언어 | TypeScript |
| 차트 라이브러리 | ECharts (echarts-for-react) |
| 데이터 처리 | PapaParse, lodash, dayjs |
| 배포 | Vercel |

## 선정 이유

### React + Vite

SPA 제작을 위한 프레임워크 중에서 React가 생태계가 가장 넓다. 차트 라이브러리·UI 라이브러리·데이터 처리 도구 대부분이 React를 기준으로 만들어져 있어서, 우리가 필요한 도구를 못 찾는 상황이 발생할 확률이 가장 낮다. 또한 Claude Code가 가장 숙달된 프레임워크이기 때문에, 바이브코딩으로 구현하는 맥락에서 생성 품질과 속도가 안정적이다. 에러가 났을 때 검색으로 해결책을 찾을 확률도 가장 높다.

Vite는 React를 빠르게 개발·빌드하기 위한 도구로, 개발 서버 기동과 핫 리로드가 가볍고 설정이 단순하다.

### TypeScript

레포 구조의 레이어 설계(Types → Schema → DataLoader → Analysis → Visualization → Layout)가 타입 시스템을 전제로 한다. `src/types/` 레이어는 TypeScript 없이는 존재 이유가 약해지고, 레이어 간 의존성을 `verify-structure.sh`로 검증하는 것도 타입 기반이어야 안정적이다.

또한 Claude Code가 생성하는 코드의 품질은 TypeScript 환경에서 더 안정적이다. 타입 에러를 컴파일 시점에 감지하고 자가 교정할 수 있기 때문이다.

### ECharts

투자 대시보드 맥락에서 차트 라이브러리의 선택이 결과물의 인상을 크게 좌우한다. ECharts를 선택한 이유는 두 가지다.

첫째, 투자 데이터에 자주 쓰이는 차트(캔들스틱, 히트맵, 박스 플롯 등)가 기본 제공된다. Recharts 같은 대안에서는 이런 차트를 직접 구현해야 하는 부담이 생긴다.

둘째, 기본 디자인이 세련된 편이라 별도 커스터마이징 없이도 완성도 있는 인상을 준다. 해커톤의 실용성·창의성 평가에서 미세하게 유리하다.

번들 크기가 큰 편이라는 단점이 있지만, 해커톤 규모의 배포에서는 문제가 되지 않는다.

### PapaParse, lodash, dayjs

데이터 처리 계층은 표준 조합을 그대로 따른다. PapaParse는 CSV 파싱의 사실상 표준이고, lodash는 데이터 변환 유틸리티의 표준, dayjs는 가볍고 API가 직관적인 날짜 라이브러리다. 이 조합은 논란의 여지가 없는 선택이라 별도의 비교 검토가 불필요하다.

### Vercel

배포 플랫폼은 "심사 기간 동안 안정적으로 접속 가능한가"가 최우선 기준이다. Vercel은 GitHub 연결 기반의 자동 배포가 가장 단순하고, 무료 플랜으로 해커톤 규모의 트래픽을 충분히 감당한다. 설정에서 시간을 까먹을 위험이 가장 적다.

Netlify·Cloudflare Pages도 유사한 대안이지만, Vercel의 기본값이 가장 안정적이라는 점에서 우선된다.

## 고려했으나 선택하지 않은 대안

- **Vue, Svelte:** 각각 장점이 있으나, Claude Code의 생성 품질과 생태계 규모에서 React가 우위에 있다. 해커톤에서 표준에서 벗어나는 선택은 예상치 못한 시간 소모로 이어질 위험이 있다.
- **Next.js:** SSR·SSG까지 지원하는 풀 프레임워크로, 순수 SPA를 목표로 하는 우리 설계에는 과한 도구다.
- **Recharts:** React 전용 차트 라이브러리로 React 통합은 더 자연스럽지만, 투자 데이터에 필요한 고급 차트를 직접 구현해야 하는 부담이 있다.
- **D3.js, Visx:** 유연성은 가장 높지만 러닝 커브와 구현 시간이 커서 바이브코딩 맥락에 부적합하다.
