# 21-frontend-aesthetics.md

## 개요

이 스킬은 투자 데이터 대시보드에 적용될 시각적 규범을 정의한다. 어떤 
투자 데이터가 들어와도 일관된 시각 언어로 표현되도록 보장하는 것이 목적이다.

### 왜 이 스킬이 필요한가

가이드 없이 LLM에게 대시보드 UI를 생성시키면 학습 데이터의 통계적 중심값으로 
수렴한다. Inter 폰트, 흰 배경 위 파스텔 그라디언트, 균일한 간격, 
무난한 둥근 모서리 같은 "안전한" 선택들이다. 이 일반화된 미학은 두 가지 문제를 
낳는다.

첫째, AI 생성물임이 한눈에 드러나 제품의 정체성을 훼손한다. 둘째, 금융 UI에 
필요한 정보 위계·숫자 가독성·관례적 색 규약을 무시한다. 투자 사용자는 
"독특한" 인터페이스가 아니라 "익숙한 관례 안에서 정보를 빠르게 읽는" 
인터페이스를 기대한다.

이 스킬은 그 관례를 결정 가능한 규칙으로 고정한다.

### 다른 Skills와의 관계

이 스킬은 `13-render.md`가 "어떤 차트를 고를지"와 "어느 슬롯에 무엇을 
바인딩할지"를 결정한 이후에 적용된다. 그 차트의 폰트·색·간격·모션을 어떻게 
렌더링할지가 이 스킬의 영역이다. 레이아웃 자체는 Figma 시안이 고정하므로 
배치 결정은 별도 스킬이 아닌 13-render의 슬롯 바인딩이 담당한다.

차트 선택 규칙과 독립적으로 동작하므로, 차트 종류나 레이아웃이 바뀌어도 
시각 규범은 유지된다. 이 독립성이 이 스킬을 "미적 감각"이 아닌 "규칙 엔진"으로 
만든다.

### 적용 범위와 검증

이 스킬의 규칙은 `src/visualization/`과 `src/layout/` 레이어의 구현에서 
강제된다. 각 규칙 섹션(색상·타이포그래피·모션·배경)은 끝에 **금지 목록**을 
포함한다. 이 스킬 파일이 금지 목록의 **단일 진실의 원천**이며, 
`verify-skills.sh`는 스킬 파일의 금지 목록 섹션을 파싱하여 실제 `src/` 코드에서 
해당 항목이 사용되지 않았는지 검사한다. 리스트를 수정하려면 이 스킬 파일만 
고치면 된다. 규칙 위반은 빌드 실패의 원인이 된다.

## 색상

### 원칙

색은 장식이 아니라 **의미 전달 장치**다. 금융 UI에서 색은 문자보다 먼저 읽히는 
신호이므로, 의미와 색의 매핑은 일관되고 관례적이어야 한다. 평범해 보이는 것을 
두려워 말되, 자의적으로 "창의적인" 색 선택을 하지 않는다.

대시보드는 배경·카드·기본 텍스트가 전체 면적의 대부분을 차지하는 무채색 기반이 
되어야 하며, 의미 색은 필요한 순간에만 날카롭게 등장한다. 
dominant 무채색과 sharp accent의 조합이 기본이고, 균등하게 배분된 소심한 
팔레트는 정보 위계를 무너뜨린다.

### 의미 색 체계 (5색)

다음 다섯 가지 의미에 색을 귀속시킨다. 각 의미는 "계열"로 고정하되, 구체 명도·채도는 
맥락에 따라 선택한다.

**긍정 / 상승 — 녹색 계열**
- 적용: 수익률 상승, 긍정적 시장 심리, 실적 개선, 목표 달성
- 사용 대상: 숫자 앞 "+" 기호, 상승 화살표, 긍정 뱃지 테두리·텍스트
- 주의: 녹색이 "환경·건강" 같은 무관 맥락에 쓰이지 않도록. 금융 맥락에서 녹색은 
  항상 상승·긍정을 의미한다.

**부정 / 하락 — 빨강 계열**
- 적용: 수익률 하락, 부정적 시장 심리, 실적 악화, 리스크 경고
- 사용 대상: 숫자 앞 "-" 기호, 하락 화살표, 부정 뱃지 테두리·텍스트, 알림 배지
- 주의: 경고·에러와 구분될 필요가 있다면 에러는 더 진한 빨강, 하락은 중간 명도.

**중립 / 정보 — 파랑 계열**
- 적용: 정보성 라벨, 상태 표시(극단적이지만 방향이 모호한 심리 상태 포함), 
  링크, 하이라이트
- 사용 대상: 보조 텍스트 중 강조가 필요한 것, 인사이트 불릿, 링크
- 주의: "정보"와 "긍정"은 다르다. 파랑이 긍정 자리를 침범하지 않도록.

**강조 / 주의 환기 — 노랑 또는 골드 계열**
- 적용: 숫자 중 가장 중요한 한두 개(점수, 핵심 지표값), 주의 요구 항목
- 사용 대상: 큰 수치 디스플레이, 주의 아이콘
- 주의: 남발하지 않는다. 한 화면에 두세 곳 이하.

**기본 / 구조 — 무채색 계열 (슬레이트·회색)**
- 적용: 배경, 카드, 구분선, 본문 텍스트, 보조 텍스트
- 사용 대상: 전체 UI의 70~80%
- 주의: 옅은 회색 위 옅은 회색은 금지 (대비 부족)

### AI slop 회피 — 금지 목록

학습 데이터 중심값에서 유래하는 "AI slop" 색 패턴과 금융 UI에서 특히 거슬리는 
패턴을 금지한다.

**금지 그라디언트**
- 보라 → 분홍 그라디언트 (AI 생성물의 시그니처)
- 청록 → 자주 그라디언트
- 파스텔 무지개 그라디언트
- 배경 전체를 채우는 수직 그라디언트

**금지 색 조합**
- 흰 배경 + 옅은 보라 강조
- 네온 계열 포화색 남발 (에너지 드링크 UI 느낌)
- 검정 배경 + 네온 녹/분홍 (게이밍 UI 느낌, 금융에 부적합)

**금지 사용 패턴**
- 의미 색을 장식으로 사용 (상승·하락과 무관한 곳에 녹색·빨강 남발)
- 같은 면적에 세 가지 이상 의미 색이 동시 등장
- 대비비 4.5:1 미만의 텍스트·배경 조합 (접근성 최소선)

### 톤과 무드

대시보드의 전반 무드는 **차분한 무채색 기반 + 의미 있는 순간의 채도 있는 액센트**다. 
영감을 얻을 수 있는 참고 방향:

- Bloomberg Terminal의 어두운 배경 + 주황·청록 액센트
- TradingView의 흰 배경 + 녹/빨 캔들 + 회색 구조
- Notion·Linear의 최소주의 무채색 + 좁은 의미 색 팔레트

이 중 어느 하나를 베끼지 않되, "구조는 무채색, 의미는 컬러"의 원칙을 공유한다.

## 타이포그래피

### 원칙

대시보드는 숫자가 주인공이다. 일반 웹의 "예쁘고 독특한" 폰트 기준은 금융 UI에 
그대로 적용되지 않는다. 타이포 선택은 다음 세 가지 요구사항 위에서 결정된다.

**숫자 가독성.** 모든 숫자의 가로폭이 동일해야 세로 정렬이 맞는다 
(tabular figures). 0과 O, 1과 l, I가 혼동되지 않아야 한다. 소수점과 
쉼표가 또렷이 구분되어야 한다.

**정보 밀도.** 좁은 공간에 많은 정보가 놓인다. x-height가 적당히 커서 
작은 크기에서도 읽히는 폰트가 유리하다. 장식적 곡선이 많으면 
밀집 환경에서 시각 소음이 된다.

**긴 응시 가능성.** 사용자는 대시보드를 오래 본다. 과도한 개성을 가진 
폰트는 초반엔 흥미롭지만 지속적 응시에서 피로를 유발한다. 
차분하고 균형 잡힌 서체가 금융 UI의 기본이다.

### 폰트 스택

**주 폰트: Wanted Sans**

영·한을 단일 폰트로 처리한다. Wanted Sans는 확장 라틴과 한글을 모두 
지원하며, 기하학적 명료함과 인간적 균형을 함께 갖는다. 
한·영 혼재 화면에서 자소 높이와 굵기가 어긋나는 fallback 문제가 발생하지 
않는다.

```css
font-family: 'Wanted Sans Variable', 'Wanted Sans', sans-serif;
```

**대안 폰트: Pretendard**

Wanted Sans 사용이 어려운 경우(CDN 장애, 라이선스 제약 등) Pretendard로 
대체한다. 자소 철학이 유사하여 교체 시 UI 톤이 크게 깨지지 않는다.

```css
font-family: 'Pretendard Variable', 'Pretendard', sans-serif;
```

두 폰트 외의 선택은 이 스킬의 허용 범위를 벗어난다.

### 역할 분할

대시보드의 폰트 역할은 **3개 이하**로 유지한다. 그 이상은 시각적 혼란을 
야기한다.

**1. UI 본문·라벨 (필수).** Wanted Sans Regular (400) 또는 Medium (500). 
카드 제목, 표 헤더, 본문 텍스트, 버튼 라벨에 사용.

**2. 숫자·지표값 (필수).** 별도 monospace 폰트를 두지 않는다. 대신 
Wanted Sans의 tabular-nums OpenType 기능을 CSS로 활성화한다:

```css
font-variant-numeric: tabular-nums;
```

이 속성은 숫자를 표시하는 모든 요소(지표 카드, 표 셀, 차트 축 라벨 등)의 
기본값이다. Wanted Sans 자체가 지원하면 탭형 숫자로 렌더링되고, 지원이 
불완전하더라도 레이아웃이 깨지지 않는다.

**3. 제목·브랜드 (선택).** 서비스 로고와 상단 대제목에만 제한적으로 
사용. 본문과 구별하기 위해 weight로만 대비하되 (Wanted Sans 800/900), 
별도 폰트 패밀리를 추가하지 않는다.

### 숫자 표현 규칙

금융 UI에서 숫자는 텍스트가 아니라 데이터다. 다음 규칙을 지킨다.

- **우측 정렬.** 표와 카드의 숫자는 우측 정렬이 기본. 세로로 쌓인 숫자의 
  자릿수가 시각적으로 맞아야 한다.
- **천 단위 구분.** `1,234,567`처럼 쉼표를 사용. 공백이나 점 구분자는 
  사용하지 않는다.
- **소수점 자릿수 고정.** 같은 컬럼·카드 내 숫자는 소수점 자릿수를 
  통일. `4.25%`와 `4.3%`가 섞이면 정렬이 깨진다.
- **부호 표기.** 양수는 명시적으로 `+` 기호 표기 (예: `+2.36$`). 
  0은 기호 없이 `0.00`. 음수는 `-`.
- **단위 축약.** 큰 숫자는 K·M·B·T 축약 사용 (예: `2.8B`). 한글 
  맥락에서는 "억"·"조" 사용 가능하되, 같은 화면에서 둘을 혼용하지 
  않는다.

### weight와 크기 대비

대비는 사용하되 극단적 대비는 피한다. 금융 UI는 랜딩 페이지가 아니다.

**weight 대비.**
- 본문: 400 (Regular)
- 강조 라벨·카드 제목: 500 (Medium) 또는 600 (SemiBold)
- 큰 숫자 디스플레이: 700 (Bold)
- 극단적 디스플레이 타이틀: 800~900 (선택적, 브랜드 요소에만)

100/200 같은 thin weight는 숫자 가독성을 무너뜨리므로 본문·숫자에 
사용하지 않는다.

**크기 대비.**
- 가장 작은 텍스트: 12px (보조 라벨, 캡션)
- 본문: 14px
- 카드 제목: 16px
- 섹션 제목: 20~24px
- 메인 숫자 디스플레이: 32~48px
- 브랜드·히어로 타이틀: 최대 60px (선택적)

작은 텍스트에서 큰 숫자까지 약 2~3배 점프가 권장 리듬이다. 5배 이상 점프는 
시각적 분절감이 과해 금융 UI에 부적합하다.

**금지 폰트 (대시보드 일반 사용)**

  (에디토리얼 맥락에 부적합)

  (숫자 가독성 저하)

**금지 사용 패턴**
- 한 화면에 서로 다른 폰트 패밀리 3개 이상 혼용
- thin weight (100/200)로 숫자 표시
- 소수점 자릿수가 통일되지 않은 숫자 정렬
- 텍스트 전체에 `text-transform: uppercase` 적용 (한글에 영향 없지만 
  영어 혼재 시 가독성 저하)
- 자간(letter-spacing) 극단적 적용 (-0.05em 이하 또는 0.1em 이상)

## 모션

### 원칙

금융 대시보드의 모션은 **최소한이어야 하지만 0이어서는 안 된다**. 
완전히 정적인 UI는 AI 생성물의 시그니처이고, 반대로 과한 모션은 
정보 접근을 지연시킨다. 대시보드의 모션은 세 가지 원칙을 따른다.

**집중형.** 모션을 여기저기 분산하지 않는다. 소수의 결정적 순간에 
잘 설계된 모션 하나가, 곳곳에 흩뿌려진 micro-interaction보다 
효과적이다.

**기능적.** 모션은 장식이 아니라 정보다. "무언가 바뀌었다", "무언가 
반응했다", "무언가 준비 중이다"를 전달하기 위해 존재한다. 이유 없는 
모션은 제거한다.

**빠르다.** 금융 사용자는 정보를 빨리 보고 싶다. 페이지 로드 시 
요소들이 엇갈려 등장하는 연출(staggered reveals)은 금융 UI에 
부적합하다. 모션은 0.3초 이내에 끝난다.

### 모션의 집중 대상

대시보드에서 모션은 두 종류의 순간에만 강하게 등장한다.

**1. 데이터 업데이트 플래시.**

숫자가 변할 때 그 숫자의 배경을 0.3초간 의미 색으로 반짝이게 한다. 
상승은 녹색, 하락은 빨강. 사용자가 화면을 잠깐 보지 않고 있다가 
돌아왔을 때 "어느 값이 방금 바뀌었는지" 한눈에 알게 하는 것이 목적이다.

- 지속 시간: 300ms
- easing: `ease-out`
- 구현: CSS `@keyframes`로 background-color 전환. React state 변화에 
  key 속성을 연결해 재트리거.

이 플래시는 금융 UI의 시그니처 모션이다. 생략하면 "움직이지 않는 
숫자판" 인상이 강해져 AI slop에 가까워진다.

**2. 차트 첫 렌더 애니메이션.**

차트가 처음 그려질 때 선이 왼쪽에서 오른쪽으로 그려지거나, 바가 
바닥에서 올라오는 효과. ECharts는 이 애니메이션을 기본 제공한다.

- 지속 시간: 500~800ms (ECharts 기본값 사용)
- easing: `cubicOut` (ECharts 기본)
- 구현: ECharts 옵션의 `animation: true` 확인. 별도 커스텀 불필요.

같은 데이터로 차트를 다시 렌더할 때는 애니메이션을 생략하거나 
더 짧게 (`animationDurationUpdate`: 300). 사용자가 필터를 바꾸는데 
매번 차트가 처음부터 그려지면 피로감이 쌓인다.

### 일반 CSS 관례 수준의 모션

아래 항목은 "스킬이 특별히 규정하지 않아도 모든 웹앱이 기본으로 
가지는" 수준의 모션이다. 이 스킬은 이 영역을 상세히 다루지 않되, 
다음 가이드라인만 남긴다.

- **hover 효과:** transition 100~150ms. `transition: all 120ms ease;` 
  수준.
- **클릭 피드백:** transform scale 0.98 또는 배경색 순간 어두움. 
  100ms 이내.
- **뷰 전환:** 필요 시 fade 100~150ms. 과한 slide·swipe 전환은 피한다.

### 기술 제약

**CSS와 ECharts 내장 애니메이션만 사용한다.**

- CSS transitions·keyframes로 해결 가능한 것은 CSS로.
- 차트 내부 애니메이션은 ECharts 옵션으로.
- Motion 라이브러리(framer-motion, Motion One 등) 추가 설치는 금지. 
  현재 대시보드 요구사항은 CSS로 충분히 커버된다.
- Lottie, rive 등 외부 애니메이션 자산 추가도 금지.

### 금지 목록

**금지 모션 패턴**
- 페이지 첫 로드 시 여러 요소가 엇갈려 등장하는 staggered reveals
- 스크롤 트리거 애니메이션 (대시보드는 보통 스크롤이 짧거나 없음)
- 장식용 반복 애니메이션 (끊임없이 회전하는 아이콘, 맥동하는 
  그라디언트 등 — 로딩 스피너 제외)
- 페이지 전환 시 slide, flip, cube 같은 3D 전환
- parallax 효과

**금지 지속 시간**
- 500ms를 넘는 hover·클릭 피드백
- 1초를 넘는 페이지 전환
- 데이터 업데이트 플래시 500ms 초과 (사용자가 다음 행동을 하기 전에 
  끝나야 함)

**금지 기술**
- framer-motion, Motion One 등 외부 애니메이션 라이브러리
- GSAP, anime.js 등 JS 기반 타임라인 라이브러리
- Lottie, rive 등 외부 애니메이션 자산
- CSS `animation` 속성으로 무한 반복 (`iteration-count: infinite`) — 
  로딩 인디케이터 한정 예외

## 배경

### 원칙

대시보드는 **정보 캔버스**다. 배경은 그 캔버스 위에 놓인 데이터를 
빛나게 만드는 무대이지, 스스로 주연이 되어선 안 된다. 동시에 완전히 
납작한 흰색 배경은 "정적이고 생기 없다"는 인상을 주며, 이는 AI 
생성물의 전형적 신호다.

배경은 두 가지 균형을 잡아야 한다. **공간감은 있되 조용해야 하고**, 
**구조는 드러내되 장식을 더하지 말아야 한다**.

### 배경 색 계층

완전 단색 흰 배경 대신 세 단계의 명도를 사용하여 자연스러운 레이어 
감을 만든다.

**바디 배경 (가장 낮은 레이어).**
- 색: 미묘하게 따뜻하거나 차가운 무채색. `slate-50` 또는 `gray-50` 
  수준.
- 역할: 전체 캔버스. 카드들이 이 위에 떠 있는 인상을 만듦.
- 완전한 `#FFFFFF`는 피한다.

**카드 배경 (중간 레이어).**
- 색: 순백 `white` (`#FFFFFF`) 또는 극히 옅은 오프화이트.
- 역할: 정보를 담는 컨테이너. 바디 배경보다 밝아 시선을 끎.

**보조 요소 배경 (구분선·입력창·호버 상태 등).**
- 색: `slate-100`, `gray-100` 수준.
- 역할: 같은 카드 내에서도 하위 구획을 나눌 때 사용.

이 3단계 명도 차이만으로도 그림자 없이 깊이감이 생긴다.

### 카드 구분: 테두리 vs 그림자

카드를 바디 배경과 구분하는 방법은 두 가지이며, **둘 중 하나만 선택한다.**

**옵션 A — 테두리 중심.**
- `border: 1px solid slate-200` 수준의 얇은 경계선.
- 그림자는 없거나 거의 없음 (`shadow-none` 또는 매우 약한 
  `shadow-sm`).
- 느낌: 깔끔하고 차분함. 문서스러움. 블룸버그 터미널 계열.

**옵션 B — 그림자 중심.**
- 테두리는 없음 또는 투명.
- 약한 그림자 `shadow-sm` (`0 1px 2px rgba(0,0,0,0.05)` 수준).
- 느낌: 부상감 있음. 모던함. Notion·Linear 계열.

둘 다 쓰면 카드가 "테두리 + 그림자 + 배경 색 차이"까지 3중으로 
구분되어 시각적 소음이 된다. 한 화면 안에서는 하나의 옵션을 일관되게 
사용한다.

**선택 기준:** 대시보드의 전반 무드에 따라. 진지하고 정보 밀도가 
높다면 옵션 A, 친근하고 여유 있다면 옵션 B.

### 그라디언트 사용 규칙

색상 섹션에서 이미 **대형 색상 그라디언트**(보라→분홍 등)를 금지했다. 
배경 섹션에서는 **허용되는 그라디언트의 범위**를 명시한다.

**허용: 미묘한 명도 그라디언트.**
- 무채색 계열 내에서, 거의 보이지 않을 정도의 변화.
- 예: 바디 배경에 `from-slate-50 to-gray-50` 수준의 수직 그라디언트.
- 목적: 완전 단색의 "납작함"을 덜고 공간감을 부여.
- 대비 한계: 시작색과 끝색의 명도 차이가 5% 이내.

**금지: 색상 그라디언트.**
- 색상 섹션에서 이미 금지된 패턴(보라-분홍, 청록-자주, 
  파스텔 무지개, 수직 전체 채움 등)을 다시 확인.
- 금융 대시보드에서 "배경이 알록달록한" 인상을 주면 정보 신뢰도가 
  떨어진다.

### 패턴·장식 효과 제한

금융 대시보드에서는 배경 장식을 거의 사용하지 않는다.

**금지되는 배경 장식.**
- 반복 패턴 (도트, 격자, 빗금 등)
- 이미지 배경
- 블러 처리된 추상 도형
- 파티클 효과, 애니메이션 배경

**예외적으로 허용되는 경우.**
- 빈 상태(empty state)의 일러스트 — "아직 데이터가 없습니다" 같은 
  영역에 한정된 작은 일러스트는 허용.
- 로고·브랜드 요소가 들어가는 헤더 영역 — 다른 규칙(타이포·색)을 
  따르는 한도 내에서 약간의 장식 가능.

### 금지 목록

**금지 배경 색**
- 순백 단색 `#FFFFFF`만으로 채워진 바디 배경 (카드는 허용)
- 포화도 높은 배경색 (예: `blue-500` 수준의 채도로 바디 배경 채움)
- 어두운 배경 + 밝은 카드 조합 (이번 스킬은 라이트 모드 기준)

**금지 장식**
- 반복 패턴 (geometric patterns 포함)
- 이미지 배경
- 블러된 추상 도형 (glassmorphism 배경)
- 애니메이션되는 배경 요소

**금지 그라디언트**
- 색상 섹션의 금지 그라디언트 목록을 참조
- 명도 차이가 큰 그라디언트 (시작과 끝의 명도 차가 10% 초과)

**금지 중복**
- 카드에 테두리와 그림자를 동시에 강하게 적용
- 바디 배경과 카드 배경의 명도 차이가 미미해 카드 경계가 불분명한 
  상태

## 다른 Skills와의 관계

이 스킬은 투자 대시보드의 시각 규범을 담당한다. 다른 Skills 파일들과 
함께 작동하며, 각자의 책임 경계가 명확히 설정되어야 한다.

현재 다른 Skills 구성: 기획팀 4개(`00-assumptions`, `01-data-profile`, 
`02-data-analysis`, `03-insight`)와 개발팀 3개(`11-ingest`, `12-analyze`, 
`13-render`). 이 스킬은 13-render 가 산출한 차트 spec 위에 시각 속성을 
적용하는 단계이며, 우선순위 충돌이 발생하면 본 스킬의 §금지 목록이 
다른 모든 결정을 이긴다.

**현재 확정된 원칙 두 가지:**

**1. 개입 시점.** 이 스킬은 차트 종류·레이아웃·지표 선택이 결정된 
이후에 적용된다. "무엇을 보여줄지"가 아니라 "결정된 것을 어떻게 
렌더링할지"를 규정한다.

**2. 의미 색의 우선.** 다른 스킬이 색을 지정하더라도, 이 스킬의 의미 
색 체계(상승 녹, 하락 빨, 정보 파랑, 강조 노랑, 구조 무채색)가 
우선한다. 의미 색은 금융 UI의 관례이며 사용자 정보 인지의 기반이 
되므로, 장식이나 미적 선호로 덮어쓸 수 없다.

세부 경계 설정은 추후 작업.
